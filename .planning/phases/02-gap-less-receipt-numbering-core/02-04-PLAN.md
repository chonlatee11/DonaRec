<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| concurrent callers → counter row | Many txs race for the same counter; FOR UPDATE must serialize them |
| caller rollback → counter/ledger | Rollback must undo the number atomically (no phantom ledger row, no consumed gap) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-10 | Tampering | duplicate receipt number under concurrency | mitigate | FOR UPDATE serialization proven by >=50-parallel test; UNIQUE backstop proven to fire (Task 1) |
| T-02-11 | Tampering | gap created by rollback (SEQUENCE-style) | mitigate | Counter+ledger rollback together; rollback-reuses-number proven (Task 2) |
| T-02-12 | Tampering | concurrent first-allocation race on new fiscal year | mitigate | InitCounterRow ON CONFLICT DO NOTHING + relock proven safe by TestAllocator_ConcurrentNewYear (Pitfall 1) |

| T-02-SC | Tampering | npm/pip/cargo installs | accept | errgroup/pgerrcode already present in go.mod/go.sum (no new external package; `go get golang.org/x/sync` only promotes an existing indirect dep) |
</threat_model>
