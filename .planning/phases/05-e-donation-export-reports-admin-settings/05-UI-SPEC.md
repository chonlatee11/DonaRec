# Phase 5 — UI Design Contract

> Visual and interaction contract for Phase 5: e-Donation Export, Reports & Admin Settings.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> **Note on frontend state:** `donnarec-web/` is fully established (Phase 3 bootstrap, Phase 4
> Admin Settings). This phase does **not** re-bootstrap the design system — it **extends** it
> with three new staff-facing surfaces: (1) an e-Donation Export + Keyed-Status Tracking screen
> (Checker/Admin only), (2) a Donation Summary Reports screen (all staff), and (3) new nav items
> in `AppShell`. Backup/Restore (NFR-08) is an ops runbook + cron companion container — it has
> **zero frontend footprint** and is intentionally out of scope for this contract (see note at
> the end of Screen Interaction Contracts). All tokens below are inherited unchanged from
> 03-UI-SPEC.md / 04-UI-SPEC.md unless explicitly marked "new in Phase 5".
>
> Despite the roadmap heading "...& Admin Settings", the Admin Settings UI itself (templates/
> signature/number-format) was delivered in Phase 4 (Screen 6) per CONTEXT.md's explicit scope
> note — this phase only extends the Phase 4 config store with e-Donation field-mapping and
> aging-threshold config values (D-75), which are **not** a new UI surface (planner may add a
> fifth tab to the existing `SettingsTabs`/Screen 6, reusing that contract as-is — no new visual
> spec required for that addition).

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn (already initialized — `donnarec-web/components.json`) |
| Preset | Default slate base — blue primary (inherited, unchanged) |
| Component library | Radix UI (via shadcn) |
| Icon library | Lucide React |
| Font (Thai, app UI) | Sarabun — `next/font/google`, weights 400 + 600 (inherited) |
| Font (Latin, app UI) | Inter — `next/font/google`, weights 400 + 600 (inherited) |
| Monospace (masked IDs, receipt numbers) | `font-mono` — system stack (inherited) |

**shadcn components to install (Phase 5 addition):**

```bash
npx shadcn@latest add checkbox
```

`checkbox` is the one new component this phase needs — for the aging table's bulk multi-select
(header "select all" with indeterminate state + per-row selection, D-67). This is an **official
shadcn registry** component, not the native-`<input type="checkbox">` pattern `ConsentBlock` used
for its single always-visible consent checkbox — a data table with tri-state select-all benefits
from Radix's built-in `indeterminate` support and consistent focus/keyboard handling.

All other components (`button`, `input`, `select`, `label`, `badge`, `table`, `dialog`,
`alert`, `alert-dialog`, `card`, `separator`, `tooltip`, `pagination`, `calendar`, `popover`,
`skeleton`, `toast`, `tabs`) are already installed from Phase 3/4 and reused as-is.

No third-party registries. shadcn official registry only.

## Spacing Scale

Inherited unchanged from 03/04-UI-SPEC.md (multiples of 4 only):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gap, badge padding |
| sm | 8px | Field-to-field vertical gap (compact), button horizontal padding |
| md | 16px | Default field-to-field gap, card body padding |
| lg | 24px | Section header-to-content gap, tab panel padding, gap between stat cards |
| xl | 32px | Between form sections, page content top padding |
| 2xl | 48px | Major section breaks (e.g., between filter bar and table) |
| 3xl | 64px | Page-level vertical padding on layout shell |

**Exceptions (Phase 5 additions):**
- Minimum interactive touch target: **44px** (inherited, applies to all new buttons/checkboxes).
- Table row height: **56px** minimum (inherited from Phase 3 donation list), reused for the
  aging table.
- Stat card grid (aging bucket cards, report summary cards): 3-column grid on desktop
  (≥768px), gap **lg (24px)**, stacking to single column below 768px.
- Bulk action bar: fixed **48px** height, appears directly above the table header when ≥1 row
  is selected — same height token as the "2xl major section break" value, used here as a
  height rather than a gap so it doesn't visually compete with the table's own header row.

## Typography

Inherited unchanged from 03/04-UI-SPEC.md — exactly 4 sizes, exactly 2 weights. No new
size/weight is introduced this phase; report "big numbers" reuse the existing Page Heading
style rather than inventing a fifth "display" size.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Caption / Label | 14px | 400 | 1.4 | Filter labels, table headers, badge text, stat-card labels, record-count preview |
| Body | 16px | 400 | 1.5 | Table cell content, dialog body text |
| Subheading | 20px | 600 | 1.3 | Card section headings, tab section titles, modal titles |
| Page Heading | 28px | 600 | 1.2 | Page `<h1>` (e.g., "ส่งออก e-Donation", "รายงานสรุปการบริจาค"); **also reused for report summary card figures** (total amount / total count / average) so they read as the visually dominant number on the card |

**Phase 5 additions (no new sizes/weights — reuse existing scale):**
- Record-count preview text ("พบ {n} รายการที่ตรงเงื่อนไข"): 14px weight 400, slate-600.
- Aging deadline date ("เดดไลน์: 5 ส.ค. 2569"): 14px weight 400, `font-mono` for the date portion
  (consistent with the existing receipt-number/masked-ID mono convention for structured data).
- Report breakdown table amount columns: 16px weight 400, right-aligned, `Intl.NumberFormat`
  comma-formatted (same convention as Screen 1's donation amount column).

### Sensitive-Action Confirmation (Export — not destructive, but PII-heavy)

Export is **not** a destructive action (nothing is deleted, D-74 stream-only), so it does not
use the `destructive` button variant — but because the file contains full plaintext 13-digit
national/tax IDs, it follows the same **audited-reveal confirmation pattern** Phase 3 established
for "เปิดเผยข้อมูล" (PII reveal), using a shadcn `AlertDialog`:

**ส่งออกข้อมูล e-Donation (Export confirmation)**
- Dialog title: ยืนยันการส่งออกข้อมูล e-Donation
- Warning banner (amber-50 background, amber-600 icon+text, inside the dialog body):
  "ไฟล์นี้มีเลขประจำตัวผู้เสียภาษี/บัตรประชาชนแบบเต็ม 13 หลักของผู้บริจาค {n} ราย การส่งออกจะถูก
  บันทึกในระบบ audit พร้อมชื่อผู้ดำเนินการ วันเวลา และช่วงข้อมูลที่เลือก กรุณาจัดเก็บและลบไฟล์
  อย่างปลอดภัยหลังใช้งาน ห้ามอัปโหลดไปยังระบบอื่นที่ไม่ปลอดภัย"
- Confirm: "ยืนยันการส่งออก" (accent)
- Cancel: "ยกเลิก" (outline)
- On confirm: file streams directly to the browser (D-74 — no server-side copy persists);
  success toast: "ส่งออกข้อมูลเรียบร้อยแล้ว — {n} รายการ"

### Bulk Action Feedback

- Mark success toast: "ทำเครื่องหมายว่าคีย์เข้า e-Donation แล้ว {n} รายการ"
- Unmark success toast: "ยกเลิกการทำเครื่องหมาย {n} รายการ"

## Screen Interaction Contracts

Screens 1–6 are inherited unchanged from 03-UI-SPEC.md / 04-UI-SPEC.md. This phase adds
Screen 7 and Screen 8, plus two new `AppShell` nav items.

### Screen 7: e-Donation Export & Keyed-Status Tracking (new)

**Route:** `/e-donation` — Checker/Admin only (RBAC gate + nav-item visibility both gated,
same pattern as Screen 6's Admin gate but for the Checker-OR-Admin role pair — reuse the Go
`RequireAnyRole(Checker, Admin)` OR-guard already fixed in Phase 3's bug #3, and add a parallel
`isCheckerOrAdminViewer()` helper alongside the existing `isAdminViewer()` in
`lib/session-role.ts` to drive the nav-link UX hint — the Go guard remains the real authority).

**Layout:** `Tabs` component (reused from Phase 4), two tabs, persistent page header
"ส่งออก e-Donation" (Page Heading, 28px/600) above the tabs.

**Tab A — ส่งออกข้อมูล (Export):**

Filter bar (horizontal, single row on desktop — reuses `DonationFilterBar`'s Calendar Popover
date-range pattern from Phase 3):
- Date range: วันที่ออกใบเสร็จ ตั้งแต่ / ถึง (two Calendar Popover fields, defaults to current
  fiscal month)
- Keyed-status `Select`: ทั้งหมด / ยังไม่คีย์ / คีย์แล้ว (default: ยังไม่คีย์ — the common export
  workflow is exporting what hasn't been keyed yet)
- Record-count preview (14px slate-600, live-updates as filters change): "พบ {n} รายการที่ตรง
  เงื่อนไข"

Export buttons row, directly below the filter bar:
- "ส่งออก Excel (.xlsx)" (accent, Lucide `FileSpreadsheet` icon)
- "ส่งออก CSV" (outline, Lucide `FileText` icon)
- Both buttons open the Export confirmation `AlertDialog` (Copywriting Contract above) before
  triggering the actual download.
- During generation: button shows a spinner + "กำลังสร้างไฟล์..." and is disabled (decrypting
  many records can take a moment); the button that was NOT clicked remains disabled too, to
  avoid a second concurrent export request.
- Zero-record filter state: both export buttons `disabled`, with the "no records" error copy
  shown as inline helper text below the button row instead of allowing a confirm-dialog-then-
  empty-file round trip.

**Tab B — ติดตามสถานะคีย์ (Keyed-Status Tracking / Aging):**

Three clickable stat cards at the top (`AgingStatCards`), 3-column grid (desktop) / stacked
(mobile), each showing a bucket count and acting as a table filter toggle:

```
┌─ ยังไม่ถึงกำหนด ─┐  ┌─ ใกล้ครบกำหนด ─┐  ┌─ เกินกำหนด ─┐
│      12          │  │       3         │  │      1       │
│   (green badge)   │  │  (amber badge)  │  │ (red badge)  │
└───────────────────┘  └─────────────────┘  └──────────────┘
```
- Card figure: Page Heading style (28px/600) in the bucket's badge text color.
- Clicking a card filters the table below to that bucket (`aria-pressed` toggle state); a
  second click (or a visible "ทั้งหมด" ghost link) clears the filter back to showing all
  unkeyed buckets together.
- Cards only ever show **unkeyed** records' bucket counts — once a record is marked keyed it
  drops out of all three buckets (the aging view's entire purpose is surfacing what's NOT yet
  done).

Table (`AgingTable`, wraps the existing `DonationTable` column-definition pattern):

| Column | Width | Notes |
|--------|-------|-------|
| ☑ (select) | 40px | shadcn `Checkbox`; header checkbox is tri-state (all/none/indeterminate) |
| วันที่ออกใบเสร็จ | 120px | `issued_at`, BE calendar display |
| ชื่อผู้บริจาค | flexible | `donor_name` |
| เลขที่ใบเสร็จ | 140px | `receipt_formatted`, `font-mono` |
| เดดไลน์คีย์ | 140px | "5 {เดือนถัดจากเดือนออกใบเสร็จ}" per D-68, `font-mono` date |
| กลุ่ม | 140px | Aging bucket `Badge` (green/amber/red) |
| สถานะคีย์ | 140px | Keyed-status `Badge` (blue "คีย์แล้ว" / slate "ยังไม่คีย์") + inline
  per-row toggle button for single-record mark/unmark without using bulk-select |

Bulk action bar (appears directly above the table, **only when ≥1 row selected**, 48px height):
```
เลือกแล้ว {n} รายการ    [ทำเครื่องหมายว่าคีย์แล้ว]  [ยกเลิกการทำเครื่องหมาย]  [ล้างการเลือก]
```
- "ทำเครื่องหมายว่าคีย์แล้ว" (accent): enabled when selection includes ≥1 not-yet-keyed row.
- "ยกเลิกการทำเครื่องหมาย" (outline): enabled when selection includes ≥1 already-keyed row
  (only meaningful if the keyed-status filter in Tab A / this tab's own status filter is set
  to show keyed records too — Tab B's table defaults to unkeyed-only per the stat cards' scope,
  so this button is mostly relevant when an admin explicitly widens the view).
- "ล้างการเลือก" (ghost): clears selection, does not affect data.
- Every mark/unmark writes an audit row per record (D-67) — no additional confirmation dialog
  is required for this action (unlike Export) since it's a reversible boolean toggle, not a
  PII-disclosure event.

Pagination: 20 rows per page, shadcn `Pagination` (reused from Screen 1).

### Screen 8: Donation Summary Reports (new)

**Route:** `/reports` — visible to **all staff** (Maker/Checker/Admin), no RBAC gate on the
nav item or route (D-71: no PII in this report, transparent to everyone).

**Layout:** Single column, max-width full (this is a data-dense reporting screen, unlike the
680px-capped donation form). Page heading "รายงานสรุปการบริจาค" (28px/600) at top.

**Filter bar** (reuses the Calendar Popover date-range pattern):
- Date range: ช่วงเวลา ตั้งแต่ / ถึง (two Calendar Popover fields, default: current fiscal year)
- "แสดงรายงาน" (accent) — explicit apply button, consistent with Screen 1's Search/Clear
  pattern (not a live-update-on-change filter, to avoid re-querying aggregates on every
  keystroke/date-pick)
- "ล้างการกรอง" (ghost)

**Summary cards** (`ReportSummaryCards`, 3-column grid desktop / stacked mobile), directly
below the filter bar:

```
┌─ ยอดบริจาครวม (บาท) ─┐  ┌─ จำนวนใบเสร็จ (ใบ) ─┐  ┌─ ยอดเฉลี่ยต่อใบ (บาท) ─┐
│    1,234,567.89        │  │        87              │  │       14,190.44          │
└─────────────────────────┘  └────────────────────────┘  └──────────────────────────┘
```
- Card label: 14px slate-600, above the figure.
- Card figure: Page Heading style (28px/600), slate-900 — deliberately **not** accent-colored;
  these are informational, not interactive, and accent stays reserved for actionable elements
  per the 10% budget.
- Figures use `Intl.NumberFormat('th-TH')` comma formatting (consistent with Screen 1's amount
  column).

**Breakdown table** (`ReportBreakdownTable`), below the summary cards:
- Group-by toggle: two-option segmented control (same pattern as Phase 4's HTML/Real-PDF
  preview toggle) — "รายเดือน" (default) / "รายวัน"
- Columns: ช่วงเวลา (period label, e.g. "ต.ค. 2569" or "1 ต.ค. 2569") | จำนวนใบเสร็จ (right-
  aligned integer) | ยอดรวม (บาท) (right-aligned, comma-formatted)
- No pagination cap needed for monthly grouping (max 12 rows/year); daily grouping paginates
  at 31 rows/page (reuse existing `Pagination`).

**Export row**, below the breakdown table:
- "ส่งออกรายงาน Excel (.xlsx)" (outline) + "ส่งออกรายงาน CSV" (outline) — **no confirmation
  dialog** (unlike Screen 7's export) since this export contains **zero PII** (D-70).

## Accessibility Contract

Inherited unchanged from 03/04-UI-SPEC.md, plus (Phase 5 additions):

- Aging table header checkbox: `aria-label="เลือกทั้งหมด"`, uses Radix `Checkbox`'s native
  indeterminate state (`aria-checked="mixed"`) when some but not all visible rows are selected.
- Per-row checkbox: `aria-label="เลือกรายการ {receipt_formatted}"` (never a bare "select" label
  repeated identically across rows).
- Aging stat cards (clickable filter toggles): `role="button"`, `tabIndex={0}`,
  `aria-pressed="{true|false}"` reflecting the active bucket filter, keyboard-activatable
  (Enter/Space).
- Bulk action bar selection-count text: `aria-live="polite"` (announces "เลือกแล้ว {n} รายการ"
  as selection changes, without needing focus to move).
- Export `AlertDialog`: focus trap + return focus to the triggering export button on close,
  same pattern as Phase 3's PII-reveal and cancel dialogs.
- Export/report generation loading state: `aria-busy="true"` on the button, spinner has
  `aria-hidden="true"` (the "กำลังสร้างไฟล์..." text label carries the accessible meaning).
- Report summary card figures: `aria-label` includes full context, e.g.
  `aria-label="ยอดบริจาครวม 1,234,567.89 บาท"` (not just the bare number, for screen readers).
- Keyed-status / aging-bucket badges: `aria-label` with full Thai text (e.g.,
  `aria-label="สถานะคีย์: คีย์เข้า e-Donation แล้ว"`, `aria-label="กลุ่มเดดไลน์: เกินกำหนด"`),
  same convention as the existing `StatusBadge`.
