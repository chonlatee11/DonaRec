## Phase Goal

**As a** hospital operations user (Checker/Admin/Maker), **I want to** have the shared data model, config store, and file-streaming substrate for e-Donation export, keyed-status tracking, and reporting in place, **so that** the export/aging/report slices can be built on a proven, no-deploy-configurable foundation without reinventing storage or file encoding.

> Substrate note: This is the shared migration + file-writer foundation the plan-phase orchestrator explicitly authorized ("group into waves respecting shared migration dependencies"). Migrations 000013/000014 and the config store are consumed by the FR-30 (export), FR-31 (keyed/aging), and FR-32 (report) slices in later waves. Each downstream slice remains a real vertical slice; this plan only provides the shared columns, queries, and streaming helper they all depend on.

<objective>
Add the shared data layer and file-streaming substrate for Phase 5: the excelize dependency, migration 000013 (keyed metadata columns on `donations`), migration 000014 (`edonation_config` single-row config table), the sqlc query set for export/aging/keyed/report/config, a reusable stream-only `.xlsx`/`.csv` writer, and the `edonation_config` accessor.

Purpose: Every later slice (export, keyed/aging, report) reads these queries and streams via this writer. Centralizing them here avoids migration-number races and duplicated file-encoding code across parallel slices.
Output: excelize in go.mod; migrations 000013/000014; `internal/db/queries/edonation.sql` + `reports.sql` (sqlc-generated Go); `internal/exportfile` writer + tests; `internal/edonation/config.go` accessor + tests.
</objective>
