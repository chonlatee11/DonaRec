<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| App role → audit_log | App may INSERT/SELECT but never UPDATE/DELETE (DB-enforced) |
| DBA → audit_log | Even a DBA tampering is detectable via hash-chain verification |
| Concurrent requests → audit_log | Serialized hash-chain link via FOR UPDATE |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-audit-01 | Tampering | audit_log row mutation | mitigate | REVOKE UPDATE/DELETE from donnarec_app role; TestAuditImmutability proves denial (D-17) |
| T-1-audit-02 | Tampering | DBA-level row edit | mitigate | SHA-256 hash-chain; VerifyChain detects any altered row + reports broken id (D-17) |
| T-1-audit-conc | Tampering | concurrent hash-chain link | mitigate | GetLastAuditRowForUpdate `FOR UPDATE` serializes prev_hash read↔insert; TestConcurrentAuditInserts -race proves zero duplicate prev_hash (Pitfall 2) |
| T-1-audit-03 | Repudiation | missing audit on action | mitigate | generic AuditMiddleware covers every mutation; audit written in same tx as mutation (no goroutine) — D-15 |
| T-1-audit-04 | Repudiation | PII reveal not logged | mitigate | isPIIRevealEndpoint forces audit on reveal GET (D-13 mechanism); TestPIIRevealAudit |
| T-1-audit-05 | Information disclosure | before/after JSON leaks PII to log | accept | before/after stored in DB audit (intended, access-restricted to Admin D-16); zap logs never include the JSON payload |
</threat_model>
