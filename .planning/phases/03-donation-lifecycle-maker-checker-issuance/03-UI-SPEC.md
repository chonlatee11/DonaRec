# Phase 3 — UI Design Contract

> Visual and interaction contract for Phase 3: Donation Lifecycle & Maker-Checker Issuance.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> **Note on frontend state:** No Next.js project exists yet. The `donnarec-web/` app is
> bootstrapped as Wave 0 of this phase. All contracts below target that new app.
> shadcn must be initialized during bootstrap (see Design System section).

---

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn (initialize with `npx shadcn@latest init` during frontend bootstrap) |
| Preset | Default slate base — blue primary. Run `npx shadcn@latest init` interactively: style=default, baseColor=slate, cssVariables=yes. |
| Component library | Radix UI (via shadcn) |
| Icon library | Lucide React (bundled with shadcn) |
| Font (Thai) | Sarabun — loaded via `next/font/google`, weights 400 + 600 |
| Font (Latin) | Inter — loaded via `next/font/google`, weights 400 + 600 |
| Font fallback | system-ui, sans-serif |
| Monospace (masked IDs) | `font-mono` — system stack (Menlo, Monaco, Consolas) |

**Why Sarabun:** Close to TH Sarabun New (the PDF/document standard) so staff see visual
consistency between the back-office UI and the printed receipts. Available on Google Fonts.
Web rendering of Thai is correct at all declared sizes.

**shadcn components to install (Phase 3 scope):**
`button`, `input`, `textarea`, `select`, `form`, `label`, `badge`, `table`,
`dialog`, `alert`, `alert-dialog`, `card`, `separator`, `tooltip`, `pagination`,
`calendar`, `popover` (for date picker), `skeleton` (loading states), `toast`.

No third-party registries. shadcn official registry only.

## Spacing Scale

Declared values (multiples of 4 only):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gap, inline badge padding |
| sm | 8px | Field-to-field vertical gap (compact), button horizontal padding |
| md | 16px | Default field-to-field gap, card body padding, form section padding |
| lg | 24px | Section header-to-content gap, sidebar padding |
| xl | 32px | Between form sections, page content top padding |
| 2xl | 48px | Major section breaks (e.g., between filter bar and table) |
| 3xl | 64px | Page-level vertical padding on layout shell |

**Exceptions:**
- Minimum interactive touch target height: **44px** (WCAG 2.5.5). All buttons, select
  triggers, and input fields must reach this minimum even when font size is 14px.
- Donation list row height: **56px** minimum to accommodate two-line content (name + status).

## Typography

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Caption / Label | 14px | 400 | 1.4 | Form labels, table column headers, badge text, helper text |
| Body | 16px | 400 | 1.5 | Form field values, table cell content, panel body text |
| Subheading | 20px | 600 | 1.3 | Card section headings, sidebar group labels, modal titles |
| Page Heading | 28px | 600 | 1.2 | Page-level `<h1>` (e.g., "รายการบริจาค", "ตรวจสอบรายการ") |

**Rules:**
- Exactly 4 sizes, exactly 2 weights — do not introduce additional sizes or weights.
- Thai and Latin text share the same size/weight declarations; Sarabun covers Thai,
  Inter covers Latin (CSS: `font-family: 'Sarabun', 'Inter', system-ui, sans-serif`).
- Masked national ID (`x-xxxx-xxxxx-1234`) displayed in `font-mono` at 14px weight 400.
- Error/validation messages: 14px weight 400, destructive color.
- Receipt number (issued state): 16px weight 600, displayed in `font-mono`.

### Destructive Confirmation Dialogs

All destructive actions require a shadcn `AlertDialog` — not a browser `confirm()`.

**ปฏิเสธถาวร (Reject Permanently)**
- Dialog title: ยืนยันการปฏิเสธรายการ
- Body: รายการนี้จะถูกปฏิเสธถาวรและไม่สามารถนำกลับมาส่งอนุมัติได้อีก
- Mandatory reason textarea: ระบุเหตุผลการปฏิเสธ *
- Confirm button: "ปฏิเสธถาวร" (destructive variant)
- Cancel: "ยกเลิก" (outline)

**ยกเลิกใบเสร็จ — Void only (edonation_keyed = false)**
- Dialog title: ยืนยันการยกเลิกใบเสร็จ
- Body: ใบเสร็จเลขที่ **{receipt_formatted}** จะถูกยกเลิก เลขที่จะยังคงอยู่ในระบบเพื่อการตรวจสอบบัญชี
- Mandatory reason textarea: ระบุเหตุผลการยกเลิก *
- Confirm: "ยืนยันยกเลิก" (destructive)
- Cancel: "ยกเลิก" (outline)

**ยกเลิกใบเสร็จ — Void when edonation_keyed = true (HIGH RISK)**
- Dialog title: ยืนยันการยกเลิก — ใบเสร็จนี้คีย์เข้า e-Donation แล้ว
- Alert banner (red-50 background, red-600 icon+text): ใบเสร็จนี้ถูกคีย์เข้าระบบ e-Donation ของกรมสรรพากรแล้ว กรุณายืนยันว่าได้ดำเนินการแก้ไข/เพิกถอนรายการฝั่งสรรพากรเรียบร้อยแล้ว ก่อนยกเลิกในระบบนี้
- Mandatory reason textarea 1: ระบุเหตุผลการยกเลิก *
- Mandatory reason textarea 2 (RD confirmation): ยืนยันว่าได้ดำเนินการฝั่งกรมสรรพากรอย่างไร *
- Confirm: "ยืนยันว่าดำเนินการ RD แล้ว และยกเลิกใบเสร็จ" (destructive)
- Cancel: "ยกเลิก" (outline)

**ยกเลิกและออกใบแทน — Void & Reissue**
- Dialog title: ยืนยันการยกเลิกและออกใบแทน
- Body: ใบเสร็จเลขที่ **{receipt_formatted}** จะถูกยกเลิก และระบบจะสร้างรายการใหม่เพื่อออกใบแทน ใบแทนต้องผ่านการอนุมัติตามปกติและจะได้รับเลขใบเสร็จใหม่
- Mandatory reason textarea: ระบุเหตุผลและข้อมูลที่ต้องแก้ไข *
- edonation_keyed = true: เพิ่ม alert banner เดียวกับ Void case ด้านบน + mandatory RD confirmation reason
- Confirm: "ยืนยัน" (destructive)
- Cancel: "ยกเลิก" (outline)

**ตีกลับแก้ไข (Return for Edit) — not fully destructive but requires reason**
- Uses shadcn `Dialog` (not AlertDialog — action is reversible)
- Dialog title: ตีกลับรายการเพื่อแก้ไข
- Body: Maker จะได้รับแจ้งให้แก้ไขและส่งใหม่
- Mandatory reason textarea: ระบุจุดที่ต้องแก้ไข *
- Confirm: "ตีกลับแก้ไข" (outline, not destructive)
- Cancel: "ยกเลิก" (ghost)

## Screen Interaction Contracts

### Screen 1: Donation List / Search

**Layout:** Full-width table with sticky header. Filter bar above table.

**Filter bar (horizontal, single row on desktop):**
- Text input: ค้นหาชื่อผู้บริจาค (placeholder: "พิมพ์ชื่อ...") — ILIKE search
- Date range: วันที่บริจาค ตั้งแต่ / ถึง — two Calendar Popover fields
- Status select: สถานะทั้งหมด / ร่าง / รอตรวจสอบ / ออกใบเสร็จแล้ว / ปฏิเสธ / ยกเลิก
- Receipt number text input: เลขที่ใบเสร็จ (exact match, FR-10)
- "ค้นหา" button (accent) + "ล้างการกรอง" ghost button

**Table columns:**
| Column | Width | Notes |
|--------|-------|-------|
| วันที่บริจาค | 120px | `donated_at` |
| ชื่อผู้บริจาค | flexible | `donor_name` |
| จำนวนเงิน (บาท) | 120px | right-aligned, comma-formatted |
| สถานะ | 140px | Badge component with status colors |
| เลขที่ใบเสร็จ | 140px | `receipt_formatted` — shown only when status=issued/cancelled; em-dash for others |
| ผู้สร้าง | 100px | `created_by` display name |
| จัดการ | 80px | "ดูรายละเอียด" link button |

**Row rules:**
- Cancelled rows: `receipt_formatted` cell shows the number in slate-500 with strikethrough + "(ยกเลิก)" suffix
- Rejected rows: no receipt number cell value (em-dash), row text color slate-500
- Maker role: "ดูรายละเอียด" on their own drafts goes to editable form; on other statuses goes to read-only view
- Checker/Admin role: all rows go to review screen

**Pagination:** 20 rows per page, shadcn Pagination component at bottom-right

**CTA:** "สร้างรายการบริจาค" button (accent, Lucide `Plus` icon) top-right above filter bar. Visible to Maker and Admin only.

- CTA changes to "ส่งรอตรวจสอบอีกครั้ง" (accent)

### Screen 4: PII Reveal Interaction

This is a role-gated, audited, inline interaction within Screen 3 (not a separate page).

**Trigger:** "เปิดเผยข้อมูล" button (secondary variant, `Eye` Lucide icon, 14px label)

**Interaction flow:**
1. User clicks "เปิดเผยข้อมูล"
2. `AlertDialog` appears:
   - Title: "เปิดเผยเลขประจำตัวผู้เสียภาษี / เลขบัตรประชาชน"
   - Body: "การดูข้อมูลนี้จะถูกบันทึกในระบบ audit log พร้อมชื่อผู้ดำเนินการและเวลา"
   - Confirm: "ยืนยันการเปิดเผย" (accent)
   - Cancel: "ยกเลิก" (outline)
3. On confirm: POST to reveal endpoint; show skeleton loader in the ID field
4. On success: replace masked value with plaintext in same position; button label changes to "ซ่อน" (`EyeOff`); `Tooltip` on field: "ข้อมูลที่เปิดเผยจะถูกซ่อนเมื่อรีโหลดหน้า"
5. On error (403 if somehow role changed): show `Toast` error "ไม่มีสิทธิ์เปิดเผยข้อมูลนี้"

**Reveal state is session-only** (client memory, not persisted). Reload returns to masked state.

## Accessibility Contract

- All form inputs have `<label>` elements with `htmlFor` (shadcn `Form` component handles this)
- Required fields marked with `*` in label and `aria-required="true"` on input
- Error messages linked via `aria-describedby` to their field
- Status badges use `aria-label` with full Thai text (e.g., `aria-label="สถานะ: รอตรวจสอบ"`)
- Masked ID field has `aria-label="เลขประจำตัว (ซ่อน)"` and masked value is `aria-hidden="false"` but marked as presentational pattern
- Destructive `AlertDialog` traps focus and returns focus to trigger on close
- Table has `role="grid"`, `<caption>`, and `<th scope="col">` for all column headers
- "เปิดเผยข้อมูล" reveal button: `aria-label="เปิดเผยเลขประจำตัวผู้เสียภาษี"` (not just icon)
- Keyboard navigation: Tab order follows visual reading order; no keyboard traps outside modals
- Loading skeletons: `aria-busy="true"` on containing element during data fetch
- Toast notifications: `role="status"` for success, `role="alert"` for errors

## i18n Contract (Thai/English)

- Default locale: `th` (Thai)
- Second locale: `en` (English)
- i18n library: `next-intl` (confirmed in CLAUDE.md)
- Locale switcher: accessible from navigation header (globe icon + current locale label)
- Thai locale: all labels, status names, button text, error messages in Thai
- English locale: all labels in English; status badge EN labels per the Color section table
- Date display: both locales use ISO date format `YYYY-MM-DD` internally; display uses `Intl.DateTimeFormat` with `th-TH` Buddhist calendar for Thai locale
- Amount display: `Intl.NumberFormat('th-TH')` for Thai locale (e.g., `1,234.00`)
- Receipt number format is locale-independent (config-driven, always shown as-is from `receipt_formatted`)
