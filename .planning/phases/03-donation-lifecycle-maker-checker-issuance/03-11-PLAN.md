<objective>
Close the detail-screen contract gap (D-R3: audit each screen's FE type vs the REAL backend response).
Today every service method returns the thin `DonationResponse` (JSON tags donor_tax_id_masked/donor_address/
donor_email/notes, created_by=UUID, replaces/replaced_by=UUID strings) — but the Phase-3 detail/review screens
(`app/donations/[id]/page.tsx`, `components/ReviewActionPanel.tsx`, `MaskedIdField`) consume a much richer
contract: `national_id_masked`, `address`, `email`, `note`, `created_by` (display name) + `created_by_id`,
`review_history[]`, `replaces`/`replaced_by` as `{id,receipt_formatted}` objects, `edonation_keyed`, and the
server-authoritative flags `viewer_is_creator`, `can_approve`, `can_return`, `can_reject`, `can_reveal_pii`.

Introduce a `DonationDetailResponse` and a single shared `buildDetailResponse(row, claims, ...)` builder used by
`GetByID` AND all mutations (Create/UpdateDraft/Submit/Approve/Return/Reject/Cancel/Reissue) so enriching one
builder aligns every screen at once. Keep national ID masked (FR-29). Extend the E2E to assert the contract.

Purpose: make the Go API the single source of truth for the detail contract (T-03-31), so 03-12/03-13 FE screens work.
Output: DonationDetailResponse + auth-flag computation + review-history/creator-name queries + E2E assertions.
</objective>
