  <behavior>
    RED (captcha): with a fake HTTP server standing in for challenges.cloudflare.com/turnstile/v0/siteverify —
    - an empty token returns a verification-failed error without any network call
    - a token the fake server answers {"success":true} returns nil
    - a token the fake server answers {"success":false} returns a verification-failed error
    - a network/timeout error (fake server closed or context cancelled) returns a verification-failed error (fail-closed, Pitfall 5) — NEVER nil
    RED (ratelimit): with a limiter configured to N=2 burst over a long window —
    - the first 2 requests from IP "1.2.3.4" pass (c.Next reached)
    - the 3rd from the same IP aborts with 429
    - a request from a different IP "5.6.7.8" still passes (per-IP isolation)
    GREEN: implement to satisfy; REFACTOR: extract the visitor-map cleanup goroutine cleanly.
  </behavior>
  <implementation>
    Add golang.org/x/time@v0.15.0 to go.mod (official Go extended-stdlib module, legitimacy verdict OK in 06-RESEARCH Package Legitimacy Audit — no human checkpoint required).
    internal/captcha/verifier.go: define `type Verifier interface { Verify(ctx, token, remoteIP string) error }` and a sentinel `ErrCaptchaFailed`, following the narrow-interface convention of mailer.EmailSender / worker.PDFRenderer.
    internal/captcha/turnstile.go: TurnstileVerifier with a secretKey (from env) and an injectable *http.Client (so tests point it at the fake server via a base-URL field or a client transport). POST form-encoded secret/response/remoteip to siteverify; decode {success, error-codes}; return ErrCaptchaFailed on empty token, transport error, decode error, or success=false. Fail-closed on every error path.
    internal/captcha/middleware.go: `VerifyTurnstile()` gin.HandlerFunc that reads the token from the multipart/form field (agree the field name with plan 03 — use `turnstile_token`), calls Verify with c.ClientIP(), and on failure AbortWithStatusJSON 400 with a distinct captcha error shape (NOT a 422 field-validation shape — Pitfall 3, the frontend keys off a distinct message). Keep the token OUT of any domain request struct.
    internal/ratelimit/middleware.go: IPRateLimiter with a sync.Mutex-guarded map[string]*rate.Limiter keyed by c.ClientIP(), a background goroutine evicting visitors unseen for a cleanup window, and a `PerIP(r rate.Limit, b int) gin.HandlerFunc` that AbortWithStatusJSON 429 (rate_limited) when !Allow(). Reuse Gin's c.ClientIP() — do not re-implement X-Forwarded-For parsing.
    internal/config/config.go: add env-driven knobs — TURNSTILE_SECRET_KEY (string, no default, never DB-stored), plus rate-limit submissions-per-window count and window duration with sane conservative defaults (5 per 10 minutes per 06-RESEARCH Open Question 2), all overridable via env like the existing WorkerConfig pattern.
  </implementation>

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-06-05 | Denial of Service | public endpoint flood | high | mitigate | Per-IP token-bucket 429 before any DB/storage or siteverify call (fail cheap); CAPTCHA as second layer (defense-in-depth, D-83) |
| T-06-06 | Tampering | forged client captcha-success flag | high | mitigate | Server-side siteverify is authoritative; no client-asserted success is ever trusted (Anti-Pattern) |
| T-06-07 | Denial of Service | siteverify network failure | high | mitigate | Fail-closed: any transport/timeout error rejects the submission (Pitfall 5) — no silent fail-open; distinct error surfaced for the frontend |
| T-06-08 | Information Disclosure | Turnstile secret leakage | high | mitigate | Env-only (TURNSTILE_SECRET_KEY), never DB-stored, never logged (zapRequestLogger already omits bodies/headers) |
| T-06-SC | Tampering | go.mod install of x/time | medium | mitigate | golang.org/x/time@v0.15.0 verified OK (official Go module, 06-RESEARCH audit); pinned exact version |
</threat_model>
