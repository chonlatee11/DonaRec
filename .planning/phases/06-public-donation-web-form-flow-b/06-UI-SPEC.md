# Phase 6 — UI Design Contract

> Visual and interaction contract for Phase 6: Public Donation Web Form (Flow B).
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> **Note on frontend state:** `donnarec-web/` is fully established (Phase 3 bootstrap,
> Phase 4/5 back-office screens). This phase adds the project's **first unauthenticated,
> public-facing surface** — everything before this phase lives behind Keycloak login. That
> forces one structural change to the existing app shell (see "Architecture Change Required"
> below) plus three new/extended screens: (1) the public donation form, (2) a staff
> pending-review queue that surfaces Flow A + Flow B side by side (FR-08), and (3) a
> responsive-navigation retrofit of `AppShell` to satisfy NFR-06 on mobile — the current
> sidebar has **zero mobile behavior** (fixed 256px, no collapse), which this phase must fix
> since NFR-06 is explicitly anchored here as "the natural completion point for full
> bilingual/responsive coverage across the now-complete UI surface" (ROADMAP.md).
>
> All tokens (spacing, typography, color, fonts) are inherited unchanged from
> 03/04/05-UI-SPEC.md unless explicitly marked "new in Phase 6" — the public form must look
> and feel like it belongs to the same product as the back-office and the PDF receipt
> (brand continuity), even though its chrome (no sidebar, no login) is necessarily different.

## Architecture Change Required (read before planning)

`app/layout.tsx` currently wraps **every** page unconditionally in `<AppShell>` (sidebar +
header + `SignOutButton`), and `middleware.ts` only forces auth on `/`, `/donations/:path*`,
`/queue/:path*` — so a new page under, e.g., `app/donate/page.tsx` would render *inside* the
authenticated sidebar shell and call `isAdminViewer()`/`isCheckerOrAdminViewer()` against an
empty session. This must be restructured, not worked around:

1. **Split the root layout.** `app/layout.tsx` keeps only `<html>`/`<body>`, font variables,
   `NextIntlClientProvider`, `AuthSessionProvider`, `Providers` (TanStack Query) — remove the
   direct `<AppShell>` wrap.
2. **Move existing authenticated pages into a route group** `app/(app)/` (`donations/`,
   `queue/`, `e-donation/`, `reports/`, `admin/`) with `app/(app)/layout.tsx` rendering
   `<AppShell>{children}</AppShell>` — route groups don't affect URLs, so `/donations` etc.
   keep their exact paths.
3. **Add a public route group** `app/(public)/donate/page.tsx` with
   `app/(public)/layout.tsx` rendering the new minimal `PublicHeader` (see Screen 9) instead
   of `AppShell` — no sidebar, no `SignOutButton`, no role checks.
4. **`middleware.ts` matcher** stays scoped to the `(app)` routes it already lists (do **not**
   add `/donate` to the auth matcher — it must stay reachable unauthenticated) and gains
   `/queue/:path*` already present; add `/e-donation/:path*`, `/reports/:path*`,
   `/admin/:path*` only if the planner confirms they were unintentionally excluded — out of
   this phase's scope to fix, noted for awareness only.

This is a structural prerequisite for every screen below, not a visual styling choice.

## Spacing Scale

Inherited unchanged from 03/04/05-UI-SPEC.md (multiples of 4 only):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gap, badge padding |
| sm | 8px | Field-to-field vertical gap (compact), button horizontal padding |
| md | 16px | Default field-to-field gap, card body padding |
| lg | 24px | Section header-to-content gap, tab panel padding |
| xl | 32px | Between form sections, page content top padding |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level vertical padding on layout shell (desktop only — see mobile exception) |

**Exceptions (Phase 6 additions):**
- Minimum interactive touch target: **44px** (inherited) — applies to the Turnstile widget's
  surrounding tap area, the mobile nav hamburger button, and every public-form control.
- **Mobile page padding override:** below the `md` breakpoint (768px), page-level vertical
  padding drops from `3xl` (64px) to `lg` (24px), and horizontal padding drops from the
  desktop `px-6` to `px-4` — the 64px desktop padding value was never designed against a
  narrow viewport and wastes ~15% of a 375px-wide screen's vertical space above the fold.
  Applies to `AppShell`'s `<main>` and the new `PublicHeader`/public form layout alike.
- Public form card: max-width **560px** (narrower than the back-office donation form's 680px
  — a public single-column form reads better slightly narrower, and it matches the width of
  the Turnstile widget's default rendering without extra whitespace either side).
- Mobile nav drawer: full-height slide-in panel, width **280px** (or `min(280px, 85vw)` on
  very narrow screens), overlay backdrop `bg-slate-900/40`.

## Typography

Inherited unchanged from 03/04/05-UI-SPEC.md — exactly 4 sizes, exactly 2 weights. No new
size/weight introduced this phase.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Caption / Label | 14px | 400 | 1.4 | Form labels, helper text, badge text, source badge |
| Body | 16px | 400 | 1.5 | Form field values, confirmation body copy |
| Subheading | 20px | 600 | 1.3 | Card section headings, modal titles |
| Page Heading | 28px | 600 | 1.2 | Public form `<h1>` ("แบบฟอร์มบริจาค"), confirmation `<h1>`, "คิวรอตรวจสอบ" |

- **Reference number** (post-submit confirmation): **16px weight 600, `font-mono`** —
  deliberately smaller/lighter-weight than the back-office **receipt number** style (20px/600
  `font-mono`, per 03-UI-SPEC.md Screen 3) so a donor can never visually mistake the reference
  number for an actual receipt number. Same font family, different size — a small but
  load-bearing distinction given D-84's explicit "reference no. ≠ receipt number" decision.
- Source badge text (Queue/List): 14px weight 400, same convention as `StatusBadge`.
- Public form section intro text ("กรอกข้อมูลผู้บริจาคเพื่อขอรับใบเสร็จ..."): 14px weight 400,
  slate-600, directly under the Page Heading.

### Mobile Navigation Retrofit — `AppShell` (new, applies to all authenticated screens)

**Problem:** `AppShell`'s `<aside className="flex w-64 ... shrink-0">` has no responsive
behavior at all — on a 375px-wide phone the fixed 256px sidebar consumes 68% of the viewport
width, leaving the main content area unusable. NFR-06 is explicitly anchored to this phase;
this is the concrete gap it must close for the back-office side.

**Contract:**
- **≥768px (`md` and up):** unchanged — fixed 256px sidebar, always visible, exactly as
  today.
- **<768px:** sidebar is `hidden` by default. Header gains a hamburger `Button` (ghost,
  `Menu` Lucide icon, `aria-label="เมนู"`) at its left edge, 44px touch target. Tapping it
  opens the **same `<aside>` markup** as a fixed-position slide-in drawer (`fixed inset-y-0
  left-0 w-[280px] z-50`, `translate-x-0` when open / `-translate-x-full` when closed,
  `transition-transform`), with a `bg-slate-900/40` backdrop `<div>` covering the rest of the
  viewport (click-to-close). Drawer content and nav links are **identical** to the desktop
  sidebar — no separate mobile-only nav data model.
- Drawer close: backdrop click, `Escape` key, or a `X` close button (top-right of the drawer,
  44px target) — focus returns to the hamburger button on close (same focus-return discipline
  as the existing `AlertDialog` patterns).
- `PublicHeader` (Screen 9/10) needs **no** equivalent drawer — it has only the
  `LocaleSwitcher`, which already fits at any width.

**Table/filter responsiveness (applies to Screens 1, 7, 8, 11):** below 768px, filter bars
(`DonationFilterBar` and the new Queue source-chip row) already `flex-wrap` (existing
Tailwind classes) — confirm this still holds at 375px during implementation. Data tables
(`DonationTable`, `AgingTable`) get a horizontal-scroll wrapper (`overflow-x-auto`) below
768px rather than a responsive column-hiding scheme — preserves every column's visibility
(no silently-hidden data in a compliance-sensitive back office) at the cost of a scroll
gesture, which is the standard, low-risk mobile-table pattern for data-dense admin UIs.

## Accessibility Contract

Inherited unchanged from 03/04/05-UI-SPEC.md, plus (Phase 6 additions):

- Public form: every field keeps the same `<label htmlFor>` / `aria-required` / error
  `aria-describedby` discipline as Screen 2 — a first-time, possibly non-technical public
  visitor depends on this more than an internal staff user does.
- `TurnstileWidget` failure states are announced via `role="alert"` (same as other inline
  field errors) — never silently disabled without an explanation.
- Mobile nav drawer: `role="dialog"` `aria-modal="true"` `aria-label="เมนูนำทาง"`, focus
  trapped while open, focus returns to the hamburger trigger on close (mirrors the existing
  `AlertDialog` focus-trap convention already used for destructive confirmations).
- Hamburger button: `aria-expanded` reflects drawer open/closed state, `aria-controls`
  pointing at the drawer's `id`.
- Reference number copy button: `aria-label="คัดลอกเลขอ้างอิงคำขอ"` (not a bare icon).
- Source filter chips: `role="group"` `aria-label="กรองตามแหล่งที่มา"`, each chip
  `aria-pressed`.
- `SourceBadge`: `aria-label` with full Thai text, e.g. `aria-label="แหล่งที่มา: จากเว็บไซต์"`
  (same convention as `StatusBadge`/`KeyedStatusBadge`).
- Public form success state (Screen 10): the swapped-in confirmation `Card` receives
  programmatic focus (`tabIndex={-1}` + `.focus()`) on mount, and is wrapped in a live region
  (`aria-live="polite"`) so a screen-reader user submitting the form is told the outcome
  without needing to re-discover the page.
