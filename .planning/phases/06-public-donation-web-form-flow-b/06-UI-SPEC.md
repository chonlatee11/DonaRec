## Architecture Change Required (read before planning)

`app/layout.tsx` currently wraps **every** page unconditionally in `<AppShell>` (sidebar +
header + `SignOutButton`), and `middleware.ts` only forces auth on `/`, `/donations/:path*`,
`/queue/:path*` — so a new page under, e.g., `app/donate/page.tsx` would render *inside* the
authenticated sidebar shell and call `isAdminViewer()`/`isCheckerOrAdminViewer()` against an
empty session. This must be restructured, not worked around:

1. **Split the root layout.** `app/layout.tsx` keeps only `<html>`/`<body>`, font variables
   (see "Dual-Theme Architecture" — root layout loads **both** font families, `NextIntlClientProvider`,
   `AuthSessionProvider`, `Providers` (TanStack Query) — remove the direct `<AppShell>` wrap.
2. **Move existing authenticated pages into a route group** `app/(app)/` (`donations/`,
   `queue/`, `e-donation/`, `reports/`, `admin/`) with `app/(app)/layout.tsx` rendering
   `<AppShell>{children}</AppShell>` — route groups don't affect URLs, so `/donations` etc.
   keep their exact paths. **No token/theme change here** — `(app)` keeps resolving the
   existing `:root`-scoped slate/blue CSS variables exactly as today.
3. **Add a public route group** `app/(public)/donate/page.tsx` with
   `app/(public)/layout.tsx` rendering the new minimal `PublicHeader` (see Screen 9) instead
   of `AppShell` — no sidebar, no `SignOutButton`, no role checks — **and** wrapping
   `{children}` in the warm-theme scope wrapper (`<div className="theme-public ...">`, see
   "Dual-Theme Architecture") that carries the cream/pine/gold token overrides and the
   Trirong/IBM Plex Sans Thai/IBM Plex Mono font variable classes.
4. **`middleware.ts` matcher** stays scoped to the `(app)` routes it already lists (do **not**
   add `/donate` to the auth matcher — it must stay reachable unauthenticated) and gains
   `/queue/:path*` already present; add `/e-donation/:path*`, `/reports/:path*`,
   `/admin/:path*` only if the planner confirms they were unintentionally excluded — out of
   this phase's scope to fix, noted for awareness only.

This is a structural prerequisite for every screen below, not a visual styling choice.

## Dual-Theme Architecture (new — read before touching any CSS/token file)

**Why two themes:** `(app)` is an internal back-office tool (Maker/Checker/Admin, authenticated,
data-dense) — it keeps the shipped slate/blue shadcn system unchanged. `(public)` is the
donation-facing brand surface (donor, unauthenticated, first-time visitor) — it adopts a warm,
trust-building cream/pine/gold identity per the locked reference sketch. These are two
different products wearing the same component library, not one theme with a few color swaps.

**The hard requirement: zero leakage.** Nothing in this phase may touch `:root` in
`app/globals.css` or `tailwind.config.ts`'s existing `colors.*` mapping — those are the live
contract for every already-shipped `(app)` screen (Phase 3/4/5). The isolation mechanism:

### 1. A second, scoped CSS variable layer (not a global override)

Add a new stylesheet, `app/(public)/public-theme.css` (imported **only** by
`app/(public)/layout.tsx`, never by the root layout or any `(app)` file), that defines a class
selector — `.theme-public` — redefining the **same shadcn semantic variable names**
(`--background`, `--foreground`, `--card`, `--popover`, `--primary`, `--primary-foreground`,
`--secondary`, `--secondary-foreground`, `--muted`, `--muted-foreground`, `--accent`,
`--accent-foreground`, `--destructive`, `--destructive-foreground`, `--border`, `--input`,
`--ring`, `--radius`) inside that class scope, with warm HSL-equivalents of the locked hex
palette (see Color section below for the exact values). This is the **same mechanism shadcn
uses for its own `.dark` class** (already present in `globals.css`) — a scoped selector that
overrides custom-property values for its descendant subtree only. Because `(app)`'s entire
render tree is never nested inside an element carrying `.theme-public`, `:root`'s slate/blue
values remain the only values `(app)` ever resolves.

`app/(public)/layout.tsx` applies the class to a single wrapping element around
`{children}` (e.g. the `<div>` that also renders `PublicHeader`):

```tsx
<div className={cn("theme-public", trirong.variable, ibmPlexSansThai.variable, ibmPlexMono.variable)}>
  <PublicHeader />
  {children}
</div>
```

**Why reuse the same variable names instead of inventing `--public-primary` etc.:** shadcn
primitives (`Button`, `Card`, `Input`, `Alert`, `Badge`, …) are hardcoded to
`bg-primary`/`text-primary-foreground`/`bg-background`/`border-border`/`ring-ring`/`rounded-lg`
Tailwind utilities, which resolve to `hsl(var(--primary))` etc. via `tailwind.config.ts`.
Scoping the **same names** under `.theme-public` means every shadcn component the public form
reuses (`Button`, `Card`, `Input`, `Textarea`, `Select`, `Form`, `Label`, `Alert`, `Toast`) picks
up the warm palette automatically — zero prop changes, zero component forks. Inventing a
parallel `public-*` token namespace would require either forking every shadcn component used on
the public form (against the "reuse shadcn primitives" convention this project has followed
since Phase 3) or hand-wiring `className` overrides on every instance (fragile, easy to miss
one and leak a slate button onto the warm form).

### 2. Extra, non-shadcn palette tokens for bespoke elements

The warm palette has hues with no shadcn semantic slot (`--pine-2` hover state, `--sage`,
`--gold` itself as a distinct-from-primary accent, `--ink`, `--line`, `--gold-wash`). These are
declared as additional custom properties **inside the same `.theme-public` scope** (not at
`:root`), consumed via Tailwind arbitrary-value utilities (e.g. `bg-[--pine-2]`,
`text-[--sage]`) or a small `public` namespace added to `tailwind.config.ts`'s `extend.colors`
(`colors: { public: { pine: 'var(--pine)', 'pine-2': 'var(--pine-2)', sage: 'var(--sage)', gold:
'var(--gold)', 'gold-wash': 'var(--gold-wash)', ink: 'var(--ink)', line: 'var(--line)' } }` →
`bg-public-pine`, `text-public-gold`, etc.). Adding this namespace to `tailwind.config.ts` is
safe (it's additive, not a mutation of the existing `colors.primary`/`colors.secondary`/etc.
mapping `(app)` depends on).

### 3. Fonts are also scoped, not global

Back-office fonts (Sarabun + Inter) stay wired exactly as today via `body`'s
`font-family: var(--font-sarabun), var(--font-inter), ...` in `globals.css` — unchanged.
The three new public fonts (Trirong, IBM Plex Sans Thai, IBM Plex Mono — all via `next/font/google`,
loaded once in the root layout like the existing fonts, since `next/font` requires a static
import at module scope) expose CSS variables (`--font-trirong`, `--font-ibm-plex-sans-thai`,
`--font-ibm-plex-mono`) that are only **applied** (via the `.variable` className) on the
`(public)` layout's wrapper div — so the variables exist globally (harmless, they're unused
custom properties until referenced) but the `font-family` rule that *consumes* them
(`.theme-public { font-family: var(--font-ibm-plex-sans-thai), ...; }`) lives inside the same
scoped `public-theme.css` block, never touching `body`'s base `font-family`.

### 4. Radius and shadow are scoped the same way

`--radius` is redefined to `18px` inside `.theme-public` (vs. the back-office's `0.5rem`/8px at
`:root`) — since `rounded-lg`/`rounded-md`/`rounded-sm` utilities all derive from `var(--radius)`
via `tailwind.config.ts`'s `borderRadius` mapping, every shadcn `Card`/`Button`/`Input` rendered
inside `.theme-public` automatically gets the warmer, softer corner radius with no component
changes. A new `--shadow-public-card` custom property (`0 14px 34px -26px rgba(23,36,31,.4)`)
is declared in the same scope for the form `Card`'s elevation (see Color/Screen 9) — back-office
`Card`s keep their flat `border`-only treatment, unchanged.

### 5. What is explicitly NOT forked

- **Spacing scale** (4/8/16/24/32/48/64, `xs`–`3xl`) is shared, unchanged, across both themes —
  this is an engineering discipline (8pt grid), not a brand choice. The reference HTML's raw
  pixel values (30px card padding, 22px gaps, etc.) are a fast-sketch approximation, **not**
  literal implementation values — map every public-surface spacing value to the nearest already-
  declared token below when building Screens 9/10 (e.g. sketch's 30px card padding → `xl` 32px;
  22px gaps → `lg` 24px).
- **Icon library** (Lucide React) — shared. The reference sketch's inline hand-drawn SVGs are
  replaced with the equivalent Lucide icon at implementation time (e.g. its shield SVG →
  `ShieldCheck`, its checkmark circle → `CheckCircle`).
- **i18n mechanism** (`next-intl`) — shared, same message-catalog pattern, new `publicDonation.*`
  namespace (see i18n Contract).
- **Component library** (Radix UI via shadcn) — shared; no public-only component library.

| Property | Value |
|----------|-------|

| Component library | Radix UI (via shadcn) — shared by both themes |

| Font (`(app)`, Thai) | Sarabun — `next/font/google`, weights 400 + 600 (inherited, unchanged) |

| Font (`(public)`, body/labels) | **IBM Plex Sans Thai** — `next/font/google`, weights 400 + 600 |
| Font (`(public)`, reference/receipt-style numbers) | **IBM Plex Mono** — `next/font/google`, weights 400 + 500 |
| Monospace (`(app)` masked IDs, receipt numbers) | `font-mono` — system stack (inherited, unchanged) |

**shadcn components to install (Phase 6 addition):** none required — `radio-group` was
considered for the language toggle but the existing `LocaleSwitcher` pattern (a single toggle
button, already used app-wide) is reused instead (see Screen 9; its `(public)` instance is
visually restyled via the warm scope, same component). All components used (`button`, `input`,
`textarea`, `select`, `form`, `label`, `card`, `separator`, `alert`, `tooltip`, `skeleton`,
`toast`, `table`, `badge`, `pagination`, `calendar`, `popover`, `tabs`) are already installed
from Phase 3/4/5 and are reused **as-is** by the public form — the warm look comes entirely
from the `.theme-public` variable scope, not from new/forked components.

No third-party shadcn registries. shadcn official registry only. **Cloudflare Turnstile** is
loaded via its own official `<script>` (`https://challenges.cloudflare.com/turnstile/v0/api.js`)
or the `@marsidev/react-turnstile` npm wrapper around that same script — this is a CAPTCHA
widget, not a shadcn/ui registry block, so it is **not** subject to the shadcn registry vetting
gate. It is a new **npm dependency** the planner should add in Wave 0 of this phase's plan;
frontend-side it is a single self-contained `TurnstileWidget` component (see Component
Inventory). Note: Turnstile renders its own branded chrome (Cloudflare attribution, checkbox
styling) that cannot be restyled to match the warm palette — expected and acceptable, same as
any third-party verification widget.

## Spacing Scale

Shared, unchanged, across both `(app)` and `(public)` (multiples of 4 only) — see "Dual-Theme
Architecture" §5 for why this is not forked:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gap, badge padding |
| sm | 8px | Field-to-field vertical gap (compact), button horizontal padding |
| md | 16px | Default field-to-field gap, card body padding |
| lg | 24px | Section header-to-content gap, tab panel padding |
| xl | 32px | Between form sections, page content top padding |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level vertical padding on layout shell (desktop only — see mobile exception) |

**Exceptions (Phase 6 additions, apply to both themes unless noted):**
- Minimum interactive touch target: **44px** (inherited) — applies to the Turnstile widget's
  surrounding tap area, the mobile nav hamburger button (`(app)` only), and every public-form
  control.
- **Mobile page padding override:** below the `md` breakpoint (768px), page-level vertical
  padding drops from `3xl` (64px) to `lg` (24px), and horizontal padding drops from the
  desktop `px-6` to `px-4` — applies to `AppShell`'s `<main>` (`(app)`) and the `(public)`
  layout's page container alike.
- **Public form card: max-width 560px** (narrower than the back-office donation form's 680px —
  a public single-column form reads better slightly narrower, and it matches the width of the
  Turnstile widget's default rendering without extra whitespace either side).
- Mobile nav drawer (`(app)` only): full-height slide-in panel, width **280px** (or
  `min(280px, 85vw)` on very narrow screens), overlay backdrop `bg-slate-900/40`.
- **`(public)` radius override:** `--radius: 18px` inside `.theme-public` (vs. `(app)`'s
  8px/`0.5rem`) — see Dual-Theme Architecture §4. Buttons on the public surface use a **pill**
  shape (`rounded-full`, not the scaled `--radius`) — a deliberate additional divergence from
  the back-office's 6–8px rounded-rectangle buttons, taken directly from the reference sketch's
  `.btn{border-radius:999px}`, reinforcing the warm/approachable brand distinction at the most
  visible interactive element.

signals, not one:** the reference number uses IBM Plex Mono at 16px/500 inside the warm
`(public)` theme; the back-office receipt number (03-UI-SPEC.md Screen 3) uses the system
`font-mono` stack at 20px/600 inside the slate `(app)` theme. Different typeface, different
size, different color system, different route group entirely — a donor literally cannot
mistake one for the other even in a screenshot with no surrounding chrome. This strengthens
D-84's "reference no. ≠ receipt number" requirement beyond what the prior single-theme spec
could achieve with a size difference alone.

## Responsive Contract — `(public)` (mobile-first, NFR-06)

**Scope note:** the reference sketch (`06-design-reference.html`) is a full marketing landing
page (hero + donate-grid + impact + footer). Per the earlier scope mapping, only its visual
language (palette/fonts/radius/shadow) and its header/"landing" chrome atop the form are in
Phase 6 scope — the marketing hero/impact/footer sections are **not** built this phase (no
FR/NFR requires them; FR-01/FR-06 cover the donation form only). "Landing sections" below means
the `PublicHeader` + page-heading/intro block that sits above the form on `/donate`, not a
separate marketing homepage. This section covers that header, the form (Screen 9), and the
confirmation (Screen 10) — the entirety of the `(public)` visual surface this phase ships.

**Mobile-first, discrete breakpoints — not `clamp()` fluid scaling.** The reference sketch uses
`clamp()` for its hero `<h1>` (`clamp(38px,5.2vw,62px)`) because a marketing hero benefits from
continuous scaling. Screens 9/10 are functional forms, not a hero — discrete, declared
breakpoint steps are used instead, for the same reason the rest of this contract declares exact
sizes rather than ranges: predictable, snapshot-testable, and checkable against the 6-dimension
gate. **The desktop values already declared elsewhere in this document (560px card, 70px header,
28px Page Heading, etc.) are the `md`-and-up tier; every value below is the mobile-first base
and the intermediate `sm` step this contract was missing.**

### Breakpoints

Reuses the project's existing Tailwind breakpoints (`tailwind.config.ts` — no new breakpoint
introduced):

| Tier | Range | Typical device | Tailwind prefix |
|------|-------|-----------------|------------------|
| Base (mobile) | 0–639px | Phones, portrait | *(no prefix — mobile-first default)* |
| `sm` | 640–767px | Large phones, small tablets portrait | `sm:` |
| `md` and up | ≥768px | Tablets landscape, desktop | `md:` — **this is the existing threshold `AppShell`'s mobile-nav retrofit already splits on; `(public)` uses the same split for consistency across the app** |

## Accessibility Contract

Inherited unchanged from 03/04/05-UI-SPEC.md for `(app)`, plus (Phase 6 additions):

- Public form: every field keeps the same `<label htmlFor>` / `aria-required` / error
  `aria-describedby` discipline as Screen 2.
- `TurnstileWidget` failure states are announced via `role="alert"`.
- Mobile nav drawer (`(app)`): `role="dialog"` `aria-modal="true"` `aria-label="เมนูนำทาง"`,
  focus trapped while open, focus returns to the hamburger trigger on close.
- Hamburger button: `aria-expanded` reflects drawer open/closed state, `aria-controls` pointing
  at the drawer's `id`.
- Reference number copy button: `aria-label="คัดลอกเลขอ้างอิงคำขอ"` (not a bare icon).
- Source filter chips (`(app)`): `role="group"` `aria-label="กรองตามแหล่งที่มา"`, each chip
  `aria-pressed`.
- `SourceBadge` (`(app)`): `aria-label` with full Thai text, e.g. `aria-label="แหล่งที่มา:
  จากเว็บไซต์"`.
- Public form success state (Screen 10): the swapped-in confirmation `Card` receives
  programmatic focus (`tabIndex={-1}` + `.focus()`) on mount, wrapped in a live region
  (`aria-live="polite"`).

### `(public)` focus ring — deliberate override of the reference sketch (WCAG fix)

**Do NOT use gold for the focus-visible outline**, despite the reference sketch specifying
`:focus-visible{outline:2.5px solid var(--gold);outline-offset:2px}`. Measured contrast:

- Gold (`#C39A44`) on cream background (`#F5F3EC`): **≈2.36:1**
- Gold (`#C39A44`) on card white (`#FCFBF6`): **≈2.53:1**

Both fail WCAG 1.4.11 (non-text contrast, requires **≥3:1**) — a real risk on a public,
often-first-time, keyboard/screen-reader-using surface, where focus visibility is load-bearing
accessibility, not decoration. **Use `--primary` (pine, `#1E4A43`) for the `(public)` focus
ring instead** — measured contrast against cream is **≈8.94:1**, comfortably passing:

```css
.theme-public :focus-visible {
  outline: 2px solid var(--pine);
  outline-offset: 2px;
  border-radius: 6px;
}
```

Gold stays reserved for the decorative/non-focus uses listed in the Color section. This is a
Claude's-discretion correction to the locked reference, made because the reference is a
marketing sketch that was never accessibility-audited, and this project treats public-form
accessibility as a first-class requirement (see original Screen 9 accessibility notes).
Flagged explicitly for the checker and for stakeholder awareness, not silently applied.
