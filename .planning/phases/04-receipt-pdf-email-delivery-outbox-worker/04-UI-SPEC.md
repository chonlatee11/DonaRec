# Phase 4 — UI Design Contract

> Visual and interaction contract for Phase 4: Receipt PDF + Email Delivery (Outbox Worker).
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> **Note on frontend state:** `donnarec-web/` already exists (bootstrapped in Phase 3) with
> shadcn initialized, the Sarabun/Inter font pairing, the slate/blue token set, and the
> `AppShell` nav shell. This phase does **not** re-bootstrap the design system — it **extends**
> it with two new surfaces: (1) an Admin-only receipt-template/compliance settings screen with
> live preview (D-58/D-61), and (2) an email-delivery status/resend/download panel added to the
> existing Donation Detail screen (Screen 3 from 03-UI-SPEC.md). All tokens below are inherited
> unchanged from 03-UI-SPEC.md unless explicitly marked "new in Phase 4".

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn (already initialized — `donnarec-web/components.json`) |
| Preset | Default slate base — blue primary (inherited, unchanged) |
| Component library | Radix UI (via shadcn) |
| Icon library | Lucide React |
| Font (Thai, app UI) | Sarabun — `next/font/google`, weights 400 + 600 (inherited) |
| Font (Latin, app UI) | Inter — `next/font/google`, weights 400 + 600 (inherited) |
| Font (Thai, PDF + template preview) | **TH Sarabun New** — bundled `.ttf`/`.woff2` asset, embedded via `@font-face`, served from the same font file used by the Chromium PDF renderer (`fonts-thai-tlwg` package / bundled asset). **New in Phase 4.** |
| Monospace (masked IDs, code/template editor, receipt numbers) | `font-mono` — system stack (inherited) |

**New font requirement (Phase 4):** the template live-preview iframe (D-61 option a) must load
**TH Sarabun New** via `@font-face`, not the Google-Fonts "Sarabun" already used for app chrome.
These are visually similar but not glyph-identical — using the wrong one defeats the purpose of
the live preview (fidelity to the real PDF). Bundle the exact `.ttf` used by the chromedp/rod
render pipeline as a static asset and reference it in the preview iframe's injected `<style>`.

**shadcn components to install (Phase 4 additions):**
`tabs` (Admin settings screen sections). All other components (`button`, `input`, `textarea`,
`select`, `form`, `label`, `badge`, `table`, `dialog`, `alert`, `alert-dialog`, `card`,
`separator`, `tooltip`, `pagination`, `calendar`, `popover`, `skeleton`, `toast`) are already
installed from Phase 3 and reused as-is.

No third-party registries. shadcn official registry only.

## Spacing Scale

Inherited unchanged from 03-UI-SPEC.md (multiples of 4 only):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gap, inline badge padding |
| sm | 8px | Field-to-field vertical gap (compact), button horizontal padding |
| md | 16px | Default field-to-field gap, card body padding, form section padding |
| lg | 24px | Section header-to-content gap, tab panel padding |
| xl | 32px | Between form sections, page content top padding |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level vertical padding on layout shell |

**Exceptions (Phase 4 additions):**
- Minimum interactive touch target: **44px** (inherited, applies to all new buttons/inputs).
- Admin settings split-pane: editor column and preview column each have a **minimum width of
  360px** before stacking to single-column on viewports under 1024px (editor on top, preview
  below, in that order — admin needs to see what they're about to preview).
- Image upload slot (letterhead/seal/signature/watermark): fixed **96px × 96px** thumbnail
  preview tile — smaller than the full-width `SlipUploadZone` from Phase 3 since these are
  small brand assets, not documents.

|------|------|--------|-------------|-------|
| Caption / Label | 14px | 400 | 1.4 | Form labels, table headers, badge text, helper text |
| Body | 16px | 400 | 1.5 | Form field values, panel body text |

**Phase 4 additions (no new sizes/weights — reuse existing scale):**

  `font-mono` for the provider message ID only (consistent with masked-ID mono treatment).
- Receipt number in the new Email Delivery panel: reuses the existing 16px/600/`font-mono`
  receipt-number style from Screen 3 (no new style).

### Error State Copy (Phase 4 additions)

| Condition | Message (TH) |
|-----------|-------------|
| Email send failed (last attempt) | ส่งอีเมลไม่สำเร็จ — {last_error}. กรุณาตรวจสอบอีเมลผู้บริจาคแล้วลองส่งอีกครั้ง |
| Resend failed | ส่งอีเมลอีกครั้งไม่สำเร็จ — กรุณาลองใหม่ภายหลังหรือดาวน์โหลด PDF แทน |
| Max retry attempts reached | ระบบส่งอีเมลอัตโนมัติครบจำนวนครั้งที่กำหนดแล้ว — กรุณากดส่งอีกครั้งด้วยตนเอง |
| PDF not yet available for download | ยังไม่มีไฟล์ PDF ให้ดาวน์โหลด — กรุณารอให้ระบบสร้างใบเสร็จให้เสร็จสิ้นก่อน |
| Template save failed (validation) | บันทึกไม่สำเร็จ — เทมเพลตมีข้อผิดพลาด กรุณาตรวจสอบตัวแปร/รูปแบบ HTML |
| Image upload rejected (magic-byte / size) | ไฟล์ไม่รองรับ — รองรับเฉพาะ JPG, PNG ขนาดไม่เกิน 2 MB |
| Real-PDF preview render failed | สร้างตัวอย่าง PDF ไม่สำเร็จ — กรุณาลองใหม่ หรือดูตัวอย่างแบบ HTML แทน |
| Settings save network error | บันทึกการตั้งค่าไม่สำเร็จ — กรุณาตรวจสอบการเชื่อมต่อและลองอีกครั้ง |

### Confirmation / Non-Destructive Action Feedback

Resend and download are **not destructive** (per D-57: resend never allocates a new receipt
number; download is read-only) — neither uses an `AlertDialog` confirmation. They follow the
same direct-action + `Toast` feedback pattern as "อนุมัติ" in Phase 3:

- Resend success toast: "ส่งอีเมลอีกครั้งเรียบร้อยแล้ว"
- Resend error toast: uses the "Resend failed" message above, `variant: destructive`
- Download: browser-native file download, no toast needed (success is visually obvious)
- Settings save success toast: "บันทึกการตั้งค่าเรียบร้อยแล้ว — มีผลกับใบเสร็จใหม่ทันที (ใบเสร็จที่ออกไปแล้วไม่เปลี่ยนแปลง)"
  — the parenthetical is load-bearing: it reassures the admin that D-56's freeze guarantee
  holds (editing the template never retroactively changes an already-issued PDF).

## Screen Interaction Contracts

Screens 1–5 are inherited unchanged from 03-UI-SPEC.md. This phase adds Screen 6 and extends
Screen 3 with a new sub-section (Screen 3b).

### Screen 3b: Email Delivery Panel (extends Screen 3 — Donation Detail View)

**Placement:** New card section in the **left panel** of Screen 3, directly below the existing
receipt-number block, visible **only when `status` is `issued` or `cancelled`** (i.e., a receipt
— and therefore a PDF/email job — exists). Not shown for draft/pending_review/rejected.

**Layout:**
```
┌─ การส่งอีเมล ──────────────────────────────────────┐
│ [DeliveryStatusBadge]  ส่งสำเร็จ                    │
│ ผู้รับ: donor@example.com                           │
│ ส่งล่าสุด: 4 กรกฎาคม 2569 14:32 · ครั้งที่ 1         │
│                                                      │
│ [ดาวน์โหลด PDF]  [ส่งอีเมลอีกครั้ง]                 │
└──────────────────────────────────────────────────────┘
```

**Field/state rules:**
- `DeliveryStatusBadge`: uses the Email delivery status badge color tokens above; `aria-label`
  always in Thai (`สถานะการส่งอีเมล: {label}`), same accessibility pattern as `StatusBadge`.
- Recipient email shown as plain 16px body text; if donor has no email on file, show the
  "no_email" empty-state copy instead of the recipient/timestamp rows.
- Last-attempt timestamp: `Intl.DateTimeFormat('th-TH-u-ca-buddhist', ...)`, same formatter
  already used in `DonationDetailView` for `formatDateTime`.
- Attempts count shown as "ครั้งที่ {n}" (attempt #n), 14px slate-600.
- Provider message ID (if present): 14px `font-mono` slate-500, on its own line, prefixed
  "รหัสอ้างอิงผู้ให้บริการ:" — collapsed under a `<details>` (same collapsed-by-default pattern
  as Screen 3's review-history accordion) to avoid cluttering the primary view.
- **Failed state:** show the "Email send failed" error copy (14px red-600) between the
  timestamp row and the button row.
- **Button row:**
  - "ดาวน์โหลด PDF" (outline, `Download` Lucide icon) — **always rendered** when a PDF exists
    (any status with a receipt number), regardless of email delivery status. Disabled + tooltip
    "ยังไม่มีไฟล์ PDF ให้ดาวน์โหลด" only in the rare window before the worker has finished
    processing the job.
  - "ส่งอีเมลอีกครั้ง" (outline, `Mail` Lucide icon) — rendered only when donor has an email on
    file. While a send is in-flight: label changes to "กำลังส่ง..." with a spinner, button
    disabled. Role-gated: visible to Checker/Admin only (same roles that can act on issued
    receipts in Screen 3's action panel) — Maker does not see a resend button.
- Cancelled receipts: panel still shows the delivery history (read-only) but the resend button
  is **not rendered** — a cancelled receipt should not be re-sent.

### Screen 6: Admin Settings — Receipt Template & Compliance Configuration (new)

**Route:** `/settings` (Admin role only — route guard + nav-item visibility both gated to Admin,
same pattern as the existing Maker/Checker/Admin conditional rendering in `ReviewActionPanel`).
Add a new nav link in `AppShell`'s sidebar: "ตั้งค่า" (`Settings` Lucide icon), rendered only
when the signed-in user's role is Admin.

**Layout:** Two-column split, persistent across tab switches (`Tabs` component controls which
editor fields show on the **left**; the **right** live-preview pane always renders the full
receipt using current in-memory edits + all other tabs' last-saved values, so the admin always
sees the complete receipt, not just the field they're touching).

- Desktop (≥1024px): left column 45% width (min 360px), right column 55% width (min 360px),
  right column preview `sticky` to viewport top minus header height (same sticky pattern as
  Screen 3's action panel).
- Below 1024px: single column, editor stacked above preview (editor first — admin edits, then
  scrolls to check the result).

**Tabs (left column):**

1. **เทมเพลตใบเสร็จ (Receipt Template)**
   - Full HTML template `<textarea>`, `font-mono` 14px/400/1.6 line-height, **minimum height
     480px**, monospace, wraps long lines (no horizontal scroll requirement for MVP; a proper
     syntax-highlighting code editor is a nice-to-have, not required this phase — planner may
     substitute a richer editor component later without changing this contract's visual/behavior
     requirements).
   - Helper text above the textarea (14px slate-600): "แก้ไข HTML ของเทมเพลตใบเสร็จ ใช้ตัวแปร
     `{{donor_name}}`, `{{amount}}`, `{{receipt_no}}` เป็นต้น" — exact placeholder token list
     supplied by planner/executor per the actual template engine chosen.
   - **Info banner (blue-50/blue-700, persistent, non-dismissible)** directly above the preview
     pane (not the editor): "ตัวอย่างนี้แสดงในกรอบจำกัดสิทธิ์ (sandbox) ไม่รันจาวาสคริปต์และไม่
     โหลดข้อมูลจากอินเทอร์เน็ต" — a direct, user-visible expression of the D-58 security mandate
     (sandboxed iframe: no JS execution, no network fetches), so admins understand why the
     preview may render slightly differently from a live browser page.

2. **รูปภาพ (Images)**
   - Four `ImageUploadSlot` instances in a 2×2 grid (single column on mobile): หัวจดหมาย/ตรา
     (letterhead/seal), ลายเซ็น (signature), ลายน้ำ (watermark) — plus a slot for a secondary
     seal image if the hospital template uses one (planner confirms exact count against the
     actual template design).
   - Each slot: 96×96px thumbnail (existing image, or empty dashed-border placeholder), label
     below, "อัปโหลดรูปภาพ" (empty state) or "เปลี่ยนรูปภาพ" / "ลบรูปภาพ" (populated state) —
     visual pattern directly reused from Phase 3's `SlipUploadZone` (dashed border, drag-drop +
     click-to-browse), just at thumbnail scale instead of full-width.
   - Accepts JPG/PNG only (no PDF — these are always raster/vector brand assets), magic-byte
     validated server-side, max 2 MB per image.

3. **ข้อความลดหย่อนภาษี (Tax Deduction Text)**
   - Textarea (16px body, not mono — this is prose, not code) for the §6 tax-deduction
     statement text (D-59: single global value, no per-donation variant).
   - A `Select` for the 1x/2x election (ตามข้อ 6 หัก 1 เท่า / หัก 2 เท่า) — single global value
     per D-59, options: "หักลดหย่อนได้ 1 เท่า" / "หักลดหย่อนได้ 2 เท่า".
   - Helper text: "ข้อความนี้ต้องได้รับการยืนยันจากฝ่ายบัญชี/กฎหมายก่อนเผยแพร่จริง" (echoes the
     ROADMAP stakeholder gate — non-blocking reminder, not a hard validation error).

4. **รูปแบบเลขที่ใบเสร็จ (Number Format)** — surfaces the existing Phase 2 config
   (separator, zero-pad digit count) in this same Admin settings screen rather than a separate
   page, per CONTEXT.md's note to consolidate with the Phase 2 config table.
   - Separator `Input` (single character, e.g. `/`)
   - Zero-pad digit count `Input` (number, e.g. `6`)
   - Live inline example directly under the two inputs, computed client-side: "ตัวอย่าง: 2569/000123"
   - Warning note (amber, inline text not a banner — low severity): "การเปลี่ยนรูปแบบมีผลกับ
     ใบเสร็จใหม่เท่านั้น ใบเสร็จที่ออกไปแล้วจะไม่เปลี่ยนรูปแบบ" (reassurance re: D-42/D-56 freeze).

**Right column — live preview pane:**
- Segmented control (two `Button` variants, not a full shadcn `Tabs` — this is a 2-option
  toggle, not navigation): "ตัวอย่างด่วน (HTML)" (default, selected) / "เรนเดอร์ PDF จริง".
- **HTML preview mode (default):** sandboxed `<iframe>` (`sandbox="allow-same-origin"` only —
  no `allow-scripts`, no `allow-popups`, no network-capable permissions), injected with the
  current in-memory template HTML + TH Sarabun `@font-face` + sample/mock donation data (never
  real PII — D-61 requirement). Debounced: template edits wait **400ms** after the last
  keystroke before re-rendering the iframe content (prevents re-render on every keystroke per
  D-61's "ห้าม re-render หนักทุก keystroke"). Shows a small inline spinner in the preview
  pane's top-right corner during the debounce/re-render window — never blocks or dims the
  editor inputs.
- **Real PDF preview mode:** clicking "เรนเดอร์ PDF จริง" triggers a server-side Chromium render
  of the current unsaved template (same pipeline as production, sample data) and displays the
  resulting PDF inline (native browser PDF viewer via `<embed>`/`<object>` or a rendered
  image, executor's choice) or offers it as a download if inline embedding isn't feasible.
  This mode is **not** debounced — it only runs on explicit button click, since it's the
  expensive/slow path. Shows a full-pane loading skeleton with the label "กำลังสร้างตัวอย่าง
  PDF..." while rendering (this can take longer than instant — set expectation, don't imply a
  frozen UI).
- Preview always uses fixture/sample donation data — **never live donor PII** (D-61 mandate).

**Save behavior:**
- "บันทึกการตั้งค่า" (accent, sticky top-right of the tab content area) saves **all tabs'**
  current values in one request (single settings object), not per-tab saves — avoids partial,
  inconsistent config states.
- On success: toast per Copywriting Contract above.
- On validation failure: inline field-level errors (14px red-600, same pattern as Phase 3 form
  validation) rather than a generic toast, so the admin can locate the problem field.
- Every save is server-audited (D-58) — no additional UI required for this; it's a backend
  concern, but the toast copy explicitly tells the admin their change is already live.

## Accessibility Contract

Inherited unchanged from 03-UI-SPEC.md, plus (Phase 4 additions):

- `DeliveryStatusBadge`: `role="status"`, `aria-label="สถานะการส่งอีเมล: {Thai label}"` — same
  pattern as `StatusBadge`.
- Sandboxed preview `<iframe>`: `title="ตัวอย่างใบเสร็จ (แซนด์บ็อกซ์)"`, and the info banner
  text above it is not purely decorative — it's the accessible explanation of why the iframe
  behaves differently from a normal page (no JS, no network).
- HTML template `<textarea>`: `<label>` + `aria-describedby` pointing to the helper text listing
  available template variables.
- Image upload slots: each has an accessible name identifying which brand asset it is (e.g.,
  `aria-label="อัปโหลดรูปลายน้ำ"`), not just a generic "Upload" label repeated four times.
- "ส่งอีเมลอีกครั้ง" button: while in the "กำลังส่ง..." (sending) state, `aria-busy="true"` and
  `aria-live="polite"` announces the status change to success/failure once the request settles.
- Number-format live example text (`aria-live="polite"`) so screen-reader users hear the updated
  example when separator/digit-count inputs change.
- Settings save success/error: reuses existing `Toast` `role="status"` / `role="alert"` pattern.

Inherited unchanged from 03-UI-SPEC.md (`next-intl`, `th` default, `en` second locale). Phase 4
adds new message namespaces to the existing catalog (`messages/th.json`, `messages/en.json`):
`settings.*` (tab labels, field labels, helper text, save/validation copy) and
`emailDelivery.*` (status labels, button labels, error copy) — mirroring the existing
`detail.*` / `actions.*` namespace conventions already in use.

- The §6 tax-deduction text and the 1x/2x election (Tab 3) are **admin-authored content**, not
  i18n catalog strings — they are stored per-locale in the config store itself (Thai text +
  English text fields side by side in the editor) since this is compliance-sensitive prose the
  hospital's accounting/legal team authors directly, not developer-maintained UI copy.
- Sample/mock preview data (Tab 1/preview pane) should include both a Thai-name and an
  English-name fixture so the admin can sanity-check the template in both `donor_language`
  branches without needing a real bilingual donor record.

| shadcn official | tabs (new); button, input, textarea, select, form, label, badge, table, dialog, alert, alert-dialog, card, separator, tooltip, pagination, calendar, popover, skeleton, toast (reused from Phase 3) | not required (official registry) |

No third-party registries declared. If a richer code-editor component (e.g., a syntax-
highlighting library) is introduced later for the HTML template textarea, it does **not** come
from a shadcn registry — it would be a plain npm dependency vetted through normal code review,
not this gate. If any shadcn third-party *registry* block is ever added, the vetting gate
(`npx shadcn view {block} --registry {url}`) must be run and results reviewed before inclusion.

## Component Inventory (Phase 4 scope)

Install the one new shadcn component:

```bash
npx shadcn@latest add tabs
```

**Custom components (new, built in Phase 4):**
- `DeliveryStatusBadge` — wraps shadcn `Badge` with email-delivery-status→color mapping
  (parallel structure to Phase 3's `StatusBadge`)
- `EmailDeliveryPanel` — Screen 3b: status badge + recipient/timestamp/attempts + resend/download
  buttons + failed-state error copy
- `SettingsTabs` — wraps shadcn `Tabs`; hosts the four Admin settings tab panels
- `TemplateEditor` — HTML `<textarea>` + debounced (400ms) sandboxed iframe live preview +
  mode toggle (HTML preview / real-PDF preview) + info banner
- `ImageUploadSlot` — single-image upload tile (96×96 thumbnail), reuses `SlipUploadZone`'s
  drag-drop/magic-byte-error visual language at a smaller scale
- `NumberFormatEditor` — separator + zero-pad digit inputs with a live-computed example string

**Reused unchanged from Phase 3:** `StatusBadge`, `Button`, `Card`, `Separator`, `Skeleton`,
`Toast`/`useToast`, `Tooltip`, form validation patterns from `DonationForm`.
