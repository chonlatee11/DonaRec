## 1. ภาพรวมและวัตถุประสงค์ (Overview)

### 1.1 วัตถุประสงค์
ระบบนี้ช่วยให้โรงพยาบาลออกใบเสร็จรับเงินบริจาคในรูปแบบ PDF ที่ถูกต้อง สามารถใช้สิทธิลดหย่อนภาษีได้ ส่งให้ผู้บริจาคทางอีเมลโดยอัตโนมัติ และจัดเก็บข้อมูลผู้บริจาคอย่างเป็นระบบเพื่อรองรับการคีย์ข้อมูลเข้าระบบ e-Donation ของกรมสรรพากร แทนกระบวนการแมนวลที่ใช้อยู่ในปัจจุบัน

### 1.2 ขอบเขต (Scope)
**อยู่ในขอบเขตเฟสนี้**
- เว็บไซต์รับบริจาคสำหรับผู้บริจาค (สร้างใหม่)
- ช่องทางให้เจ้าหน้าที่สร้างรายการบริจาคเอง
- กระบวนการตรวจสอบ/อนุมัติโดยเจ้าหน้าที่ก่อนออกใบเสร็จ
- สร้างใบเสร็จ PDF พร้อมเทมเพลต ลายน้ำ และลายเซ็นผู้มีอำนาจ
- ระบบเลขที่ใบเสร็จแบบรันต่อเนื่อง
- ส่งใบเสร็จทางอีเมล
- จัดเก็บข้อมูลผู้บริจาคตามข้อกำหนด PDPA
- เตรียม/ส่งออกข้อมูลเพื่อรองรับการคีย์เข้า e-Donation ด้วยตนเอง
- รองรับ 2 ภาษา (ไทย/อังกฤษ)

**อยู่นอกขอบเขตเฟสนี้**
- ใบเสร็จทั่วไป (non-donation) → เฟสถัดไป
- บริจาคสิ่งของ (in-kind) → ไม่รองรับ
- การเชื่อมต่อ payment gateway → ไม่มี (ปัจจุบันแมนวล 100%)
- การเชื่อมต่ออัตโนมัติกับระบบบัญชีภายในเดิม → เป็นระบบแยกต่างหาก
- การเชื่อมต่อ API ตรงกับ e-Donation กรมสรรพากร → เจ้าหน้าที่คีย์ข้อมูลเอง

### 1.3 คำจำกัดความ
| คำ | ความหมาย |
|---|---|
| ผู้บริจาค (Donor) | บุคคล/นิติบุคคลที่บริจาคเงินให้โรงพยาบาล |
| เจ้าหน้าที่ (Staff) | ผู้สร้าง/บันทึกรายการบริจาคในระบบ |
| ผู้อนุมัติ (Approver) | ผู้ตรวจสอบและอนุมัติก่อนออกใบเสร็จ |
| e-Donation | ระบบบริจาคอิเล็กทรอนิกส์ของกรมสรรพากร |
| PDPA | พ.ร.บ. คุ้มครองข้อมูลส่วนบุคคล พ.ศ. 2562 |
| สลิป | หลักฐานการโอนเงินที่ผู้บริจาคแนบมา |

## 2. ผู้ใช้งานและบทบาท (Users & Roles)

| บทบาท | สิทธิ์การใช้งานหลัก |
|---|---|
| **ผู้บริจาค** (ภายนอก) | กรอกข้อมูลบริจาค แนบสลิป ให้ความยินยอม PDPA รับใบเสร็จทางอีเมล |
| **เจ้าหน้าที่ (Maker)** | สร้างรายการบริจาคเอง / ตรวจสอบรายการจากเว็บ / แก้ไขข้อมูลก่อนอนุมัติ |
| **ผู้อนุมัติ (Checker)** | ตรวจสอบความถูกต้อง อนุมัติให้ออกใบเสร็จ หรือตีกลับ |
| **ผู้ดูแลระบบ (Admin)** | จัดการผู้ใช้/สิทธิ์ ตั้งค่าเทมเพลต ลายเซ็น เลขเริ่มต้น ดูบันทึกการใช้งาน |

> **ประเด็นต้องยืนยัน:** ผู้สร้าง (Maker) กับผู้อนุมัติ (Checker) ต้องเป็นคนละคนหรือไม่ (segregation of duties) — แนะนำให้แยกเพื่อการควบคุมภายใน

## 3. กระบวนการทำงาน (Process Flows)

### Flow A — เจ้าหน้าที่สร้างรายการเอง
1. เจ้าหน้าที่เข้าสู่ระบบ → สร้างรายการบริจาคใหม่
2. กรอกข้อมูลผู้บริจาค + จำนวนเงิน + แนบหลักฐาน
3. ส่งให้ผู้อนุมัติตรวจสอบ
4. ผู้อนุมัติ: อนุมัติ → ระบบออกเลขที่ใบเสร็จ + สร้าง PDF → ส่งอีเมล
   หรือ ตีกลับ → กลับไปขั้นแก้ไข

### Flow B — ผู้บริจาคทำรายการผ่านเว็บ
1. ผู้บริจาคเข้าเว็บบริจาค → กรอกข้อมูล + แนบสลิป + กดยินยอม PDPA
2. ระบบบันทึกเป็นรายการ "รอตรวจสอบ"
3. เจ้าหน้าที่ตรวจสอบว่ามีการบริจาคจริง (เทียบสลิป)
4. ผู้อนุมัติอนุมัติ → ออกเลขที่ใบเสร็จ + สร้าง PDF → ส่งอีเมลให้ผู้บริจาค
   หรือ ปฏิเสธ → แจ้งเหตุผล

> หลักการสำคัญ: **ทุกใบเสร็จต้องผ่านการตรวจสอบโดยมนุษย์ก่อนส่งเสมอ** — ระบบไม่ออกใบเสร็จอัตโนมัติทันทีโดยไม่มีการอนุมัติ

### 4.4 การออกเลขที่ใบเสร็จ
| รหัส | ความต้องการ | ระดับ |
|---|---|---|
| FR-15 | เลขที่ใบเสร็จอ้างอิง **ปีงบประมาณ + เลขรัน** (เช่น `2569/000123`) โดยปีงบประมาณคือรอบ **1 ต.ค. – 30 ก.ย.** ตั้งค่ารูปแบบ/ตัวคั่น/จำนวนหลักได้ | M |
| FR-16 | เลขต้อง **ไม่ซ้ำ และเรียงต่อเนื่อง ห้ามข้ามเลข** (gap-less) ภายในปีงบประมาณเดียวกัน | M |
| FR-17 | **รีเซ็ตเลขรันเป็น 1 อัตโนมัติเมื่อขึ้นปีงบประมาณใหม่** (เริ่มนับใหม่ทุกวันที่ 1 ต.ค.) | M |
| FR-18 | ระบบกำหนดปีงบประมาณจาก **วันที่ออกใบเสร็จ** โดยอัตโนมัติ (เอกสารที่ออกระหว่าง ต.ค.–ธ.ค. นับเป็นปีงบประมาณถัดไป เช่น ออกวันที่ 15 ต.ค. 2568 → ปีงบประมาณ 2569) | M |
| FR-19 | กรณียกเลิกใบเสร็จ ต้องใช้สถานะ "ยกเลิก" ไม่ลบเลขทิ้ง (เก็บร่องรอยเพื่อการตรวจสอบบัญชี/ภาษี) | M |

> **หมายเหตุทางเทคนิค:**
> - ตัวนับเลขรันต้องแยกตาม **ปีงบประมาณ** (เก็บคู่ `[ปีงบประมาณ → เลขล่าสุด]`)
> - การรับประกันเลขต่อเนื่องห้ามซ้ำในระบบ multi-user ควรออกเลขในจังหวะ commit ของฐานข้อมูล (เช่น sequence/locking หรือ atomic counter ระดับ transaction) ไม่ใช่คำนวณล่วงหน้า
> - ระวังรายการที่ "บันทึกข้ามรอบปีงบประมาณ" — ยึด **วันที่ออกใบเสร็จ (วันอนุมัติ)** เป็นเกณฑ์กำหนดปีงบประมาณให้ชัดเจน

## 6. ข้อมูลที่ต้องมีในใบเสร็จ (เพื่อใช้ลดหย่อนภาษี)

แนะนำให้ใบเสร็จมีอย่างน้อย:
- ชื่อและที่อยู่ของโรงพยาบาล + เลขประจำตัวผู้เสียภาษีของโรงพยาบาล
- เลขที่ใบเสร็จ (running) และวันที่ออก
- ชื่อ-นามสกุลผู้บริจาค + เลขประจำตัวผู้เสียภาษี/บัตรประชาชน + ที่อยู่
- จำนวนเงินบริจาค (ตัวเลขและตัวอักษร)
- วัตถุประสงค์/ประเภทการบริจาค
- ข้อความระบุสิทธิลดหย่อนภาษี (และระบุหากเข้าเงื่อนไขลดหย่อน 2 เท่า ตามประเภทสถานพยาบาล)
- ลายเซ็นผู้มีอำนาจ + ตราโรงพยาบาล

> **ประเด็นต้องยืนยันกับฝ่ายบัญชี/กฎหมายของโรงพยาบาล:** ข้อความและรูปแบบที่ถูกต้องตามประมวลรัษฎากร และเงื่อนไขการได้สิทธิลดหย่อน 1 เท่า/2 เท่า ของโรงพยาบาล

---

## 7. ความเป็นส่วนตัวและ PDPA

| หัวข้อ | ข้อกำหนด |
|---|---|
| ความยินยอม (Consent) | เก็บหลักฐานการยินยอม พร้อมวันเวลาและเวอร์ชันข้อความ |
| วัตถุประสงค์ | ใช้ข้อมูลเพื่อออกใบเสร็จและรองรับการลดหย่อนภาษีเท่านั้น |
| ระยะเวลาเก็บ | ใบเสร็จ/เอกสารภาษีมักต้องเก็บตามกฎหมายภาษี (เช่น อย่างน้อย 5 ปี) ซึ่งอาจขัดกับสิทธิขอลบของ PDPA — ต้องกำหนดนโยบายให้ชัด |
| สิทธิเจ้าของข้อมูล | รองรับการขอเข้าถึง/แก้ไขข้อมูล (สิทธิขอลบอาจถูกจำกัดโดยกฎหมายภาษี) |
| การควบคุมการเข้าถึง | จำกัดผู้เห็นเลขบัตรประชาชน/ข้อมูลอ่อนไหว ตามบทบาท |
| ความปลอดภัย | เข้ารหัสข้อมูลอ่อนไหว + บันทึกการเข้าถึง |

> **ประเด็นต้องยืนยัน:** นโยบายระยะเวลาเก็บข้อมูลที่สอดคล้องทั้งกฎหมายภาษีและ PDPA — ควรปรึกษาฝ่ายกฎหมาย/DPO ของโรงพยาบาล

## 8. ข้อจำกัดและสมมติฐาน (Constraints & Assumptions)

- ระบบเป็นส่วนเสริม **แยกต่างหาก** จากระบบบัญชีภายในเดิม (ไม่เชื่อมต่ออัตโนมัติในเฟสนี้)
- ไม่มี payment gateway — การยืนยันเงินเข้าทำโดยเจ้าหน้าที่ตรวจสลิปด้วยตนเอง
- การส่งเข้า e-Donation ทำโดยเจ้าหน้าที่คีย์เอง ระบบเพียงเตรียมข้อมูลให้
- รองรับเฉพาะเงินบริจาค (ไม่รวมสิ่งของ) ในเฟสนี้

## 9. ประเด็นที่ต้องตัดสินใจเพิ่มเติม (Open Issues)

1. ลายเซ็นดิจิทัลเป็นรูปภาพ หรือ PKI digital signature? (กระทบความน่าเชื่อถือและงบประมาณ)
2. Maker กับ Checker ต้องแยกคนหรือไม่
3. ข้อความ/รูปแบบใบเสร็จที่ถูกต้องตามสรรพากร และเงื่อนไขลดหย่อน 2 เท่า
4. นโยบายระยะเวลาเก็บข้อมูล (ภาษี vs PDPA)
5. รูปแบบไฟล์ส่งออกสำหรับคีย์ e-Donation (Excel/CSV) และฟิลด์ที่ต้องมี
6. ต้องให้ผู้บริจาคขอใบเสร็จย้อนหลัง/ออกใบแทนกรณีหายหรือไม่

## 10. ข้อเสนอแนะการแบ่งเฟส (Phasing)

**เฟส 1 (MVP)** — Back office: สร้างรายการเอง → ตรวจสอบ/อนุมัติ → ออกเลข → สร้าง PDF (เทมเพลต+ลายน้ำ+ลายเซ็น) → ส่งอีเมล + จัดเก็บข้อมูล PDPA + ส่งออกข้อมูล e-Donation

**เฟส 2** — เว็บไซต์รับบริจาคสาธารณะ (Flow B) + แจ้งสถานะผู้บริจาค + 2 ภาษาเต็มรูปแบบ

**เฟส 3** — ใบเสร็จทั่วไป (non-donation), รายงานเชิงลึก, การเชื่อมต่อระบบบัญชี/e-Donation อัตโนมัติ (ถ้าต้องการในอนาคต)

## ภาคผนวก: ตัวอย่าง User Stories พร้อม Acceptance Criteria

**US-1 — เจ้าหน้าที่ออกใบเสร็จ**
> ในฐานะเจ้าหน้าที่ ฉันต้องการสร้างและออกใบเสร็จบริจาคหลังตรวจสอบแล้ว เพื่อส่งให้ผู้บริจาคใช้ลดหย่อนภาษี
- Given รายการได้รับการอนุมัติ When ระบบออกใบเสร็จ Then ได้เลขที่รันต่อเนื่องไม่ซ้ำ และไฟล์ PDF มีเทมเพลต/ลายน้ำ/ลายเซ็นครบ
- ใบเสร็จที่ยังไม่อนุมัติจะไม่มีเลขที่และไม่ถูกส่งอีเมล

**US-2 — ผู้บริจาคทำรายการผ่านเว็บ**
> ในฐานะผู้บริจาค ฉันต้องการกรอกข้อมูลและแนบสลิปผ่านเว็บ เพื่อขอรับใบเสร็จทางอีเมล
- ต้องกดยินยอม PDPA ก่อนส่งจึงจะบันทึกได้
- หลังเจ้าหน้าที่อนุมัติ ระบบส่งใบเสร็จ PDF ไปยังอีเมลที่กรอกไว้

**US-3 — ผู้อนุมัติตรวจสอบ**
> ในฐานะผู้อนุมัติ ฉันต้องการเทียบสลิปกับยอดเงิน เพื่อยืนยันการบริจาคจริงก่อนออกใบเสร็จ
- อนุมัติได้เฉพาะเมื่อข้อมูลครบและเห็นสลิปแนบ
- ตีกลับต้องระบุเหตุผล และระบบบันทึก audit trail
